# Data Commons Mixer Architecture

*Last comprehensive review: 22.April.2026*
This document outlines how the Mixer codebase is laid out, how data flows through the system, and where to find different components.

## Notable file structures

The repository separates language-agnostic schemas from compiled application code:

*   **`proto/`**: Holds `.proto` schema definitions. 
    *   `proto/v1`, `proto/v2`, `proto/v3`: Public-facing API definitions.
    *   `proto/*.proto` (e.g., `storage.proto`, `sdmx.proto`): Internal schemas (like `SdmxDataQuery`) used for backend routing and database queries. This separates backend structs from the public API surface.
*   **`internal/proto/`**: Contains generated Go code. Files in this directory are automatically generated by `./scripts/compile_protos.sh` based on the definitions in `proto/` and should not be manually edited.

## The Macro Pipeline

Mixer uses a transcoding pipeline for incoming requests:

1.  **The HTTP Request**: External clients make HTTP/REST requests containing JSON payloads.
2.  **Envoy / ESP Transcoder**: Mixer is a gRPC application and does not natively serve HTTP. Incoming requests hitting the deployment pass through a sidecar container running Envoy (configured in `esp/`). Envoy intercepts the HTTP payload, uses Protobuf specifications (`endpoints.yaml.tmpl`), and transcodes the HTTP payload into a binary gRPC frame.
3.  **The gRPC Hand-off**: Envoy forwards the binary gRPC payload to Mixer's local port. Mixer processes this as a standard incoming gRPC call.

## Inside The Go App: `internal/server/`

The `internal/server/` directory is organized into the following functional layers:

### 1. The Edge (API Gateways)
*   **Files**: `server.go`, `handler_v1.go`, `handler_v2.go`, `handler_v3.go`
*   **Overview**: Defines the top-level gRPC server hooks. Unpacks incoming requests, performs property validation, and translates the request into an internal query before passing it to the orchestration layer.
*   **Helper Folders**: `/v0`, `/v1`, `/v2`, `/v3`. Contains version-specific helper functions and validation logic that are factored out of the main handler files.

### 2. The Orchestration Layer

*   **`/dispatcher`**: The Middleware Engine. Its primary job is to execute interceptor layers (Processors) wrapping the database request. For example, it checks Redis caches to short-circuit database queries, applies math formulas to synthesize derived statistical variables post-query, and handles traffic shadowing.
*   **`/datasources` (Plural)**: The Scatter-Gather Facade. Its only job is parallel query execution. It receives a request from the Dispatcher, fires it concurrently at every connected backend database (Local Spanner, Maps API, Remote Mixer) via a mechanism called `fetchAndMerge`, stitches the raw arrays back together, and hands the unified dataset back up.

### 3. The Storage Boundaries

*   **`/datasource` (Singular)**: The Gateway Interface. Contains the strict Go `interface` that defines the exact methods a backend driver *must* implement to qualify as a data source.
*   **`/spanner` (Concrete)**: Implements the `DataSource` interface for Google Cloud Spanner. Contains the Spanner Graph database mapped logic.
*   **`/remote` (Concrete)**: Implements the `DataSource` interface for Federation. It routes requests to an external Mixer API instead of a local database.
*   **`../../sqldb` (Concrete)**: Implements the `DataSource` interface for local SQLite files and CloudSQL instances. (Note: this package lives directly under `internal/` rather than `internal/server/`).
*   **`../../store` (Legacy)**: A standalone abstraction layer located at `internal/store/`. It manages Cloud Bigtable and BigQuery connections. This stack is invoked directly by the V1 API Edge handlers, bypassing the Dispatcher/Facade paradigm.

### 4. The Feature Domains
*   **`/stat`, `/statvar`, `/place`, `/node`, `/recon`**: Contains domain-specific business logic (e.g., calculating Place containment trees, finding relations for Statistical Variables).
*   **`/sdmx`**: Contains parsing rules, JSON-stat formatting, and constants bounded to SDMX queries.
*   **`/translator`**: Contains Natural Language processing and search translation tools.

### 5. Example Request Flow: Evaluating a V2 API Call
To understand how these routing layers collaborate, trace the execution of a standard V2 `Resolve` API request. 

1.  **(Edge)** `handler_v2.go` receives a binary gRPC payload and unpacks it into a `*pbv2.ResolveRequest`.
2.  **(The Bridge)** The Edge handler immediately evaluates `shouldDivertV2()`. This function checks startup flags (`--use_spanner_graph`) and probabilistic feature flags (`V2DivertFraction`) to determine which architectural stack the request should traverse. The pipeline splinters here:

**Path A: The Store Flow (Non-Diverted)**
*   **(Edge Fan-Out)** If the request is not diverted, the Edge handler parses the query independently and creates a concurrent Go ErrorGroup to fetch data directly from the `internal/store` (Bigtable) implementation and the Remote Mixer API.
*   **(Edge Merge)** The handler waits for both fetches to finish, merges the responses together manually, and prepares the final object.
*   **(Telemetry)** The handler now calls `maybeMirrorV3()`. This spins off a detached background thread to pass the same payload to the Spanner Graph APIs. The background thread logs latency metrics and structural diffs without blocking the live user's request.
*   **(Return)** The merged object is returned to the client.

**Path B: The Dispatcher Flow (Diverted)**
*   **(Orchestrator)** Assuming the server was booted with `--use_spanner_graph=true`, `shouldDivertV2` unconditionally returns true. The Edge handler delegates the request immediately to `dispatcher.Resolve()`.
*   **(Orchestrator)** The Dispatcher executes `PreProcess` middleware hooks (e.g. evaluating Redis caching) and hands the request down to the `datasources.go` Facade.
*   **(Storage Routing)** The Facade evaluates routing rules via `filterResolveSources()`. It inspects the request targets to determine if it should call the local database, the remote mixer, or both. Concluding it needs local data, it adds the injected `SpannerDataSource` instance to its target execution list. *(Note: The Spanner instance was injected as a Go Interface during startup via `cmd/main.go`, which constructs `datasources.NewDataSources()` and passes it into the Dispatcher).*
*   **(Storage Execution)** The Facade executes `fetchAndMerge(...)`. This function spins up parallel Goroutines to concurrently invoke the `Resolve()` method (as defined by the `DataSource` interface) on every configured data source in the target list. Because `SpannerDataSource` was placed in that list, its execution is triggered.
*   **(Orchestrator)** The Facade collects the responses from all executing drivers, merges them into a single unified dataset, and hands it back up to the Dispatcher. 
*   **(Edge)** The Dispatcher passes the fully processed result back to `handler_v2.go`, which serializes the object back to the caller.

## Server startup notes and relevant files and build files

*   **`cmd/main.go`**: Parses startup flags, establishes database connections (Spanner, Bigtable, CloudSQL, Redis), and passes dependencies to `server.go`.
*   **`internal/server/server.go`**: Maps the dependencies created by `main.go` to the active gRPC endpoints.
*   **`build/`**: Contains `Dockerfile` specifications for the Mixer application, the ESP Envoy proxy, and base images.
*   **`scripts/`**: Contains developer bash scripts (e.g., `compile_protos.sh`) that execute commands with specific CI flags.

## Deploy code

*   **`deploy/featureflags/`**: Contains YAML configuration files for each active environment (`dev.yaml`, `autopush.yaml`, `prod.yaml`, `local.yaml`). Mixer reads these at runtime to determine experimental routing and feature toggles.
*   **`build/ci/`**: Contains Cloud Build configurations (`.yaml`) used by CI/CD pipelines to build, test, and tag Docker images.

## Misc

*   **Testing (`test/` vs `*_test.go`)**: Unit tests are located adjacent to the files they test. The repository also utilizes "Golden Testing" (comparing JSON outputs against text files). The `/test` folder holds larger integration testing wrappers.
*   **Documentation (`docs/`)**: Technical and operational guides. For specific commands regarding boot sequences and profiling, see `docs/developer_guide.md`.

## Adding a New Feature: Where does it go?

General sequence for introducing a new API feature:

1.  **Public Schema:** Add definitions to `proto/[version]/`.
2.  **Internal Schema:** Add internal data structures to `proto/internal/[domain].proto`. Run `./scripts/compile_protos.sh`.
3.  **API Hook:** Register the handler method in `internal/server/handler_[version].go`.
4.  **Plumbing:** Route the request through `internal/server/dispatcher`.
5.  **DB Logic:** Implement query execution in `internal/server/spanner/`.
6.  **Feature Logic:** If the feature requires specific parsing or custom formatting, isolate that logic in a feature-specific directory (e.g., `internal/server/[my_feature]/`) rather than inlining it into the generic layers.
