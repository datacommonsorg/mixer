# Data Commons Mixer Architecture

*Last comprehensive review: 22.April.2026*
This document outlines how the Mixer codebase is laid out, how data flows through the system, and where to find different components.

## The Macro Pipeline

### Transcoding Pipeline
Mixer uses a transcoding pipeline for incoming requests:

1.  **The HTTP Request**: External clients make HTTP/REST requests containing JSON payloads.
2.  **Envoy / ESP Transcoder**: Mixer is a gRPC application and does not natively serve HTTP. Incoming requests hitting the deployment pass through a sidecar container running Envoy (configured in `esp/`). Envoy intercepts the HTTP payload, uses Protobuf specifications (`endpoints.yaml.tmpl`), and transcodes the HTTP payload into a binary gRPC frame.
3.  **The gRPC Hand-off**: Envoy forwards the binary gRPC payload to Mixer's local port. Mixer processes this as a standard incoming gRPC call.

This is how **all** external requests into Mixer get routed. Mixer does not natively serve HTTP; all REST API calls are transcoded to gRPC before they reach the Go application.

### Notable file structures
The repository separates language-agnostic schemas from compiled application code:

*   **`proto/`**: Holds `.proto` schema definitions. 
    *   `proto/v1`, `proto/v2`, `proto/v3`: Public-facing API definitions.
    *   `proto/*.proto` (e.g., `storage.proto`, `sdmx.proto`): Internal schemas (like `SdmxDataQuery`) used for backend routing and database queries. This separates backend structs from the public API surface.
*   **`internal/proto/`**: Contains generated Go code. Files in this directory are automatically generated by `./scripts/compile_protos.sh` based on the definitions in `proto/` and should not be manually edited.
*   **`build/`**: Contains `Dockerfile` specifications for the Mixer application, the ESP Envoy proxy, and base images. Also contains CI/CD configurations in `build/ci/`.
*   **`scripts/`**: Contains developer bash scripts (e.g., `compile_protos.sh`) that execute commands with specific CI flags.
*   **`deploy/`**: Contains YAML configuration files for each active environment (feature flags in `deploy/featureflags/`).

## Inside Mixer: Dispatcher Based Fulfillment

### Server Startup
Mixer initialization happens in `cmd/main.go` and `internal/server/server.go`:
*   **`cmd/main.go`**: Parses startup flags, establishes database connections (Spanner, Bigtable, CloudSQL, Redis), instantiates data sources, and configures the list of middleware processors based on flags.
*   **`internal/server/server.go`**: Maps the dependencies created by `main.go` to the active gRPC endpoints and initializes the `Dispatcher` with the configured processors and data sources.

The `internal/server/` directory is organized into the following functional layers:

### 1. The Edge (API Gateways)
*   **Files**: `handler_v1.go`, `handler_v2.go`, `handler_v3.go`
*   **Overview**: Defines the top-level gRPC server hooks. Unpacks incoming requests, performs property validation, and translates the request into an internal query before passing it to the orchestration layer.
*   **Helper Folders**: `/v0`, `/v1`, `/v2`, `/v3`. Contains version-specific helper functions and validation logic that are factored out of the main handler files.

### 2. The Orchestration Layer
This layer manages request flow, applies middleware, and coordinates parallel execution across data sources.

*   **`/dispatcher`**: The Middleware Engine. Its primary job is to execute interceptor layers (**Pre-processors** and **Post-processors**) wrapping the database request. For example, it uses Pre-processors to check Redis caches or expand remote entities, and Post-processors to apply math formulas to synthesize derived statistical variables.
    *   **`Processor` Interface**: Defined in `processor.go`, this interface is implemented by various processors scattered across the codebase (e.g., `RelationExpressionProcessor` in `/dispatcher`, `CacheProcessor` in `/redis`, and `CalculationProcessor` in `/v2/observation`).
*   **`/datasources` (Plural)**: The Scatter-Gather Facade. Its only job is parallel query execution. It receives a request from the Dispatcher, fires it concurrently at every connected backend database (Local Spanner, Maps API, Remote Mixer) via a mechanism called `fetchAndMerge`, stitches the raw arrays back together, and hands the unified dataset back up.

### 3. The Storage Boundaries
This layer defines how Mixer talks to different storage backends via a unified interface, shielding higher layers from database specifics.

*   **`/datasource` (Singular)**: The Gateway Interface. Contains the strict Go `interface` that defines the exact methods a backend driver *must* implement to qualify as a data source.
*   **`/spanner` (Concrete)**: Implements the `DataSource` interface for Google Cloud Spanner. Contains the Spanner Graph database mapped logic.
*   **`/remote` (Concrete)**: Implements the `DataSource` interface for Federation. It routes requests to an external Mixer API instead of a local database.
*   **`../../sqldb` (Concrete)**: Implements the `DataSource` interface for local SQLite files and CloudSQL instances. (Note: this package lives directly under `internal/` rather than `internal/server/`).
*   **`../../store` (Legacy)**: A standalone abstraction layer located at `internal/store/`. It manages Cloud Bigtable and BigQuery connections. This stack is invoked directly by the V1 API Edge handlers, bypassing the Dispatcher/Facade paradigm.

### 4. The Feature Domains
This layer contains business logic specific to different data domains, isolating specialized operations from the generic infrastructure.
*   **`/stat`, `/statvar`, `/place`, `/node`, `/recon`**: Contains domain-specific business logic (e.g., calculating Place containment trees, finding relations for Statistical Variables).
*   **`/sdmx`**: Contains parsing rules, JSON-stat formatting, and constants bounded to SDMX queries.
*   **`/translator`**: Contains Natural Language processing and search translation tools.

### 5. Example Request Flow: Evaluating a V2 API Call
To understand how these routing layers collaborate, trace the execution of a standard V2 `Resolve` API request traversing the modern Dispatcher flow:

1.  **(Edge)** `handler_v2.go` receives a binary gRPC payload and unpacks it into a `*pbv2.ResolveRequest`.
2.  **(The Bridge)** The Edge handler evaluates `shouldDivertV2()`. Assuming the server was booted with `--use_spanner_graph=true`, it returns true and delegates the request immediately to `dispatcher.Resolve()`.
3.  **(Orchestrator)** The Dispatcher executes `PreProcess` middleware hooks (e.g., evaluating Redis caching or expanding remote entities) and hands the request down to the `datasources.go` Facade.
4.  **(Storage Routing)** The Facade evaluates routing rules via `filterResolveSources()`. It inspects the request targets to determine if it should call the local database, the remote mixer, or both. Concluding it needs local data, it adds the injected `SpannerDataSource` instance to its target execution list.
5.  **(Storage Execution)** The Facade executes `fetchAndMerge(...)`. This function spins up parallel Goroutines to concurrently invoke the `Resolve()` method (as defined by the `DataSource` interface) on every configured data source in the target list.
6.  **(Orchestrator)** The Facade collects the responses from all executing drivers, merges them into a single unified dataset, and hands it back up to the Dispatcher. 
7.  **(Edge)** The Dispatcher passes the fully processed result back to `handler_v2.go`, which serializes the object back to the caller.

### Legacy Fulfillment Methods
While new features use the Dispatcher-based flow, some legacy endpoints (primarily V1) still bypass it:
*   **`../../store`**: Direct access to Cloud Bigtable and BigQuery, bypassing the Dispatcher/Facade paradigm.
*   Legacy handlers in `v1/` and `v0/` often contain inlined business logic and direct storage calls instead of delegating to feature domains or datasources.

#### The Store Flow (Non-Diverted V2 Path)
Even in V2, requests can still follow the legacy flow if not diverted (controlled by flags like `V2DivertFraction`):
*   **(Edge Fan-Out)** The Edge handler parses the query independently and creates a concurrent Go ErrorGroup to fetch data directly from the `internal/store` (Bigtable) implementation and the Remote Mixer API.
*   **(Edge Merge)** The handler waits for both fetches to finish, merges the responses together manually, and prepares the final object.
*   **(Telemetry)** The handler may call `maybeMirrorV3()` to spin off a detached background thread to pass the same payload to the Spanner Graph APIs for latency metrics and structural diffs without blocking the live user's request.
*   **(Return)** The merged object is returned to the client.


## Developer Guidelines

### Adding a New Feature: Where does it go?

General sequence for introducing a new API feature:

1.  **Public Schema:** Add definitions to `proto/[version]/`.
2.  **Internal Schema:** Add internal data structures to `proto/internal/[domain].proto`. Run `./scripts/compile_protos.sh`.
3.  **API Hook:** Register the handler method in `internal/server/handler_[version].go`.
4.  **Plumbing:** Route the request through `internal/server/dispatcher`.
5.  **DB Logic:** Implement query execution in `internal/server/spanner/`.
6.  **Feature Logic:** If the feature requires specific parsing or custom formatting, isolate that logic in a feature-specific directory (e.g., `internal/server/[my_feature]/`) rather than inlining it into the generic layers.

### When to use a Processor

*   **Cross-Cutting Concerns**: Use processors for logic that applies to multiple data sources or requests generally (e.g., caching, traffic shadowing).
*   **Request Augmentation**: Use `PreProcess` to discover or fetch information needed by specific data sources, without modifying the immutable request object itself.
*   **Response Synthesis**: Use `PostProcess` to modify or enrich the combined response from data sources (e.g., applying statistical formulas).
*   **Avoid Backend-Specific Coupling**: Do NOT put logic in a processor that assumes a specific storage backend unless necessary. Processors should be backend-agnostic where possible. If a task requires coordinate resolution across local and remote sources, consider if it belongs in the Facade (`datasources`) layer or a processor.

### Passing Information from Processor to DataSources via Context

Go's `context.Context` is the preferred way to pass information from a processor to individual data sources.

**Best Practice**: Always use a custom, unexported type for context keys to avoid collisions across packages.

#### 1. Define the key (e.g., in a `constants.go` or the processor file):
```go
package dispatcher

type contextKey string
const RelationExpressionExpandedEntities contextKey = "RelationExpressionExpandedEntities"
```

#### 2. Set the value in the Processor (`PreProcess`):
```go
func (p *RelationExpressionProcessor) PreProcess(rc *RequestContext) (Outcome, error) {
    dcids := []string{"geoId/06001", "geoId/06002"}
    
    // Update the context in RequestContext
    rc.Context = context.WithValue(rc.Context, RelationExpressionExpandedEntities, dcids)
    
    return Continue, nil
}
```

#### 3. Read the value in the DataSource:
```go
func (sds *SpannerDataSource) Observation(ctx context.Context, req *pbv2.ObservationRequest) (*pbv2.ObservationResponse, error) {
    // Extract the value from the context
    if remoteDcids, ok := ctx.Value(dispatcher.RelationExpressionExpandedEntities).([]string); ok {
        // Use the remoteDcids...
    }
    // ...
}
```

### Feature Flags vs. Server/CLI Flags

Mixer uses two mechanisms for configuration and feature gating: **Server/CLI Flags** and **Feature Flags**. It is important to use the correct mechanism depending on the scope of the change.

#### 1. Server/CLI Flags (Startup Flags)
*   **Definition**: Flags defined in `cmd/main.go` and parsed at startup (e.g., `port`, `config_path`). Passed to the server object during initialization.
*   **When to use**:
    *   **Infrastructure Configuration**: Setting up database connections, API ports, billing projects, or file paths.
    *   **Backend Availability**: Indicating if a specific database backend is enabled and initialized (e.g., `s.useSpannerGraph` or `s.bigtableClient != nil`).
*   **Characteristics**: Immutable after the server starts. Changing them requires a server restart.

#### 2. Feature Flags (Deployment/Release Flags)
*   **Definition**: Flags defined in `internal/featureflags/featureflags.go` and loaded from an environment-specific YAML configuration file (e.g., `V2DivertFraction`, `EnableExperimentalFeature`).
*   **When to use**:
    *   **Deployment-Specific Toggles**: Enabling/disabling features for specific environments (e.g., enabling experimental features in `autopush` or `local` environments via their respective YAML configs, but keeping them disabled in `prod`).
    *   **Traffic Control & Routing**: Toggling default execution paths (e.g., routing traffic to Spanner vs. Bigtable).
    *   **Gradual Rollouts / A/B Testing**: Supporting partial rollouts (e.g., `V2DivertFraction`) or enabling new features under a toggle.
*   **Characteristics**: Loaded at startup from a YAML config file. **Changing them requires a server restart/redeployment.** They allow environment-specific behavior to be controlled via configuration files rather than altering command-line arguments.

#### Best Practice: Combining Flags for Safe Rollouts
When introducing a new feature that relies on a new backend:
1.  Use a **Server/CLI Flag** (or server field check like `s.isBackendConfigured`) to verify the backend infrastructure is configured and initialized.
2.  Use a **Feature Flag** (like `s.flags.EnableFeature`) to control the default routing.
3.  Combine them in the routing logic: `useNewFeature := s.isBackendConfigured && s.flags.EnableFeature`. This ensures the server never routes traffic to a backend that was not initialized at startup, even if the feature flag is accidentally enabled.

### Testing and Documentation

*   **Testing (`test/` vs `*_test.go`)**: Unit tests are located adjacent to the files they test. The repository also utilizes "Golden Testing" (comparing JSON outputs against text files). The `/test` folder holds larger integration testing wrappers.
*   **Documentation (`docs/`)**: Technical and operational guides. For specific commands regarding boot sequences and profiling, see `docs/developer_guide.md`.
